# LiteShip documentation glossary

Vocabulary for prose across this repository. The public author model is define, apply, inspect; the deeper engine vocabulary remains available when a reader crosses that abstraction boundary. Technical identifiers (`Boundary`, `@liteship/core`, `liteshipMiddleware`, `--liteship-*`, `host-wired`, CLI `liteship`, ...) stay exactly as shipped.

## The shape

<!-- BEGIN DIAGRAM (canonical mental model — keep byte-identical across README / GLOSSARY / AUTHORING-MODEL; pinned by tests/unit/meta/diagram-drift.test.ts) -->

```text
defineAdaptive(...) ─▶ attrs() + plan() ─▶ explain(value)
```

- **define** — describe the input, named states, and outputs once
- **apply** — spread `attrs()` onto host markup and use the CSS from `plan()`
- **inspect** — call `explain(value)` to see the selected state, thresholds, provenance, and identity

<!-- END DIAGRAM -->

## Two-layer naming

| Layer             | Use when                                                                                                                        |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| **LiteShip**      | The one brand: product, distribution, engine, and architecture. What readers adopt, what README hooks name, what ADRs describe. |
| **`@liteship/*`** | npm namespace only: install lines, imports, package lists. Never renamed.                                                       |

**Canonical sentence:** _LiteShip — distributed as `@liteship/*` packages on npm._

## Primitives (prose register)

| Term                   | Consistent description                                                                                         |
| ---------------------- | -------------------------------------------------------------------------------------------------------------- |
| **Boundaries**         | Definitions that partition continuous inputs into named states.                                                |
| **Tokens**             | Named values and axes that belong to the design language.                                                      |
| **Styles**             | Base and named-state properties owned by one boundary.                                                         |
| **Themes**             | Coordinated token-space variants.                                                                              |
| **Compile path**       | Projection of authored intent to CSS, GLSL, WGSL, ARIA, AI, or another target artifact.                        |
| **Runtime / hot path** | Live evaluation and application of an authored definition; workers and GPU paths are explicit execution tiers. |

The beginner-facing terms are deliberately smaller:

| Term                    | Consistent description                                                                                                   |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| **Adaptive definition** | One `defineAdaptive` value that lowers through the real boundary, style, quantizer, token, theme, and compiler owners.   |
| **Named state**         | A discrete result such as `mobile`, `tablet`, or `desktop`, selected by evaluating a boundary at one input value.        |
| **Apply**               | Spread `attrs()` onto host markup and emit the matching CSS from `plan()`.                                               |
| **Inspect**             | Call `explain(value)` to retrieve the selected state, threshold trace, output provenance, capability tier, and identity. |

## Audience vocabulary policy

Vocabulary follows the reader's task instead of exposing every engine noun at the
same altitude:

| Audience                        | Allowed register                                                 | Rule                                                                                                                                          |
| ------------------------------- | ---------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Beginner authoring              | define, apply, inspect, adaptive definition, named state, output | Do not require `cast`, graph, receipt, tier, runtime-allocation, or package-fleet vocabulary to complete the first feature.                   |
| Operational CLI and diagnostics | build, check, compile, project, output, evidence, remediation    | Prefer established technical terms. A failure message must state what failed and the concrete remedy; themed synonyms cannot replace them.    |
| Expert engine and architecture  | document graph, projection, cast target, receipt, tier, kernel   | `cast` is permitted only as the verb “project to a named target”; it is not a beginner concept, noun, command synonym, or type-coercion term. |
| Historical evidence             | exact retired names when provenance requires them                | Historical text may quote an old term, but must identify it as historical and cannot define current product or API behavior.                  |

Source code, quoted external text, theatrical uses of “cast”, and TypeScript
type-cast discussion are false positives for the product term. Automated policy
checks must classify the audience and semantic use rather than globally replacing
matching text.

## Substrate and projection vocabulary (prose register)

Terms of art for the IR and projection machinery. Pin the meaning; do not translate by surface form.

Expert prose may say “Definitions cast to CSS” when naming the target explicitly.

| Term                                | Consistent description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Document graph**                  | `@liteship/core`'s content-addressed IR: a graph of typed nodes that authored definitions seal into, and that every projection target reads from. The keystone. Capitalize as a proper noun for the symbol (`DocumentGraph`); lowercase "document graph" in running prose.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Node family**                     | One of the eight node kinds a document graph holds: `signal`, `entity`, `component`, `pose`, `transition`, `projection`, `policy`, `export`. A closed union — adding one is a typed change, not a copy-paste.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Transition program**              | `@liteship/core`'s explicit multi-transition algebra (`TransitionProgram`): the composition TREE over `TransitionNode`s that replaced the per-node `routing` LABEL (ADR-0039). Four arms — **step** (one pose→pose tween, optional `delayMs`), **seq** (deterministic: total is `Σ` children + delays, each mapped to a disjoint sub-window), **par** (total is the `max` child duration; children share the window, a short one holds its final pose), **choice** (EXACTLY one branch executes, chosen by a `BranchCondition` over a named signal; the selection is an auditable receipt). Lowers via `interpretProgram` to REAL multi-offset keyframes + per-window runtime sub-samplers. Capitalize `TransitionProgram` for the symbol; "transition program" in prose. Distinct from `EdgeType`, which stays the edge flavor BETWEEN adjacent transitions the tree composes over. |
| **Motion kernel** (`sampleProgram`) | `@liteship/core`'s ONE shared motion sampler: `sampleProgram(plan, t)` returns the typed leaf value of every animated `cssVar` at normalized time `t`, unifying a composed `TransitionProgram` (per-window sub-samplers) and a flat single-tween plan behind one reader. EVERY non-CSS target samples it, and the declarative CSS `@keyframes` are generated from the SAME internal window-walk (ADR-0040, Law 4). `sampleProgramUniforms` projects a sample to the `{ css, wgsl }` `liteship:uniform-update` payload the browser floor writes and the worker posts. "Motion kernel" or the symbol `sampleProgram` in prose.                                                                                                                                                                                                                                                         |
| **Motion adapter**                  | A THIN per-target wrapper that samples the motion kernel and projects the result to that target's representation: `@liteship/scene` `MotionSampleSystem` (scene components), `@liteship/stage` `exportMotionTrack` (content-addressed video frames), `@liteship/remotion` `sampleMotionFrame` (composition style), `@liteship/worker` `motionSampleMessage` (off-thread uniform post). ADDITIVE to `@liteship/scene`'s video-crossfade `_blend`, never a merge (ADR-0040). Pinned by the differential oracle (`tests/unit/core/motion-parity.test.ts`), the reader that makes each adapter load-bearing.                                                                                                                                                                                                                                                                             |
| **Signal source**                   | The canonical vocabulary for what drives a boundary: `SignalSource` (the typed union in `@liteship/core`) is the source of truth, and `sourceToInput` / `inputToSource` round-trip its serialized input forms — the dot-string axes (`viewport.width`, `scroll.progress`, `audio.amplitude` / `audio.beat`) and the colon-delimited free-form payloads (`media:<query>`, `custom:<id>`). Every domain reads through it rather than re-parsing input strings.                                                                                                                                                                                                                                                                                                                                                                                                                         |
| **Projection target**               | A named surface a definition projects to: CSS, GLSL, WGSL, ARIA, AI manifest, or video. "Project" is the standard verb and "surface" is the emitted target.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Escalation tier** (quality tier)  | A runtime quality tier a projection may be admitted to under a policy + frame budget. `chooseTier` returns the admitted target set; a surface over budget drops to a cheaper tier. The 5-tier scale (`static`..`gpu`) is the same one the quantizer's `MotionTier` projects; see `quality-tiers.ts`. Not "rung" or "level" in prose.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **tier / tiers**                    | The capability triple resolved at the edge — `tier` (device capability), `motion`, `design` — keyed by axis. Emitted on `<html>` as `data-liteship-<axis>` and exposed typed on `Astro.locals.liteship.tiers`; the axis name is the single source (`CAP_AXES`), so the attribute and the locals field can't disagree. "tier" alone is the capability axis; "tiers" the triple.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Capsule**                         | A unit of reusable assembly built from the closed set of factory arms (ADR-0008). A property test plus a budgeted bench, not a component. "Capsule-ize" = lock a behavior as a standing capsule.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **GraphPatch**                      | The one typed mutation over a document graph: propose -> validate -> apply -> re-seal. The editor and AI proposals both mutate _through_ it; nothing edits node maps by hand.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **ValidatedProposal / ApplyToken**  | The AI-proposal security envelope. A `ValidatedProposal` is what a validator mints; the `ApplyToken` it carries is the unforgeable witness that `applyValidatedPatch` demands. Describe it as an _envelope_ or _witness_, never "permission" or "key".                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **Mutation channel**                | The client→server return leg: a remote client proposes a `GraphPatch` and the server validates it through the same refuse-seam before applying (`handleGraphMutation` in `@liteship/core`, `graphMutationRoute` in `@liteship/astro`). Three outcomes — `applied`, `refused`, `error`; a compare-and-swap on `saveGraph` makes optimistic concurrency free. The host owns the store and the route (the authority boundary). Not a general RPC layer — one validated graph-patch seam.                                                                                                                                                                                                                                                                                                                                                                                                |
| **Mutation client**                 | The client-side state half of the mutation channel: holds a current base, serializes submits, advances on `applied`, and uses `staleBase` refusals to reload through a host-owned reader before re-proposing. DOM-free rigging, not a component.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Form binding**                    | The DOM bridge from a host-authored form to the mutation client. Captures `FormData`, lets the host project it to patch ops, sets `data-liteship-mutation-state`, and emits `liteship:mutation`; the host owns markup and error rendering.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Morph-opaque**                    | A DOM subtree marked `data-liteship-morph-opaque`: the morph engine keeps the live subtree verbatim and never removes unmatched old opaque nodes. Opacity isolates diffing, not trust; new HTML is still sanitized before insertion.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **Standard Schema**                 | The V1 validation interop contract carried by `DocumentGraphNodeSchema`'s `~standard` property. It lets Standard-Schema-aware host stacks consume LiteShip's canonical node schema without a second adapter value.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **seal / address**                  | `sealNode` / `sealGraph` mint a node's content address from its canonical bytes. "Seal" = assign identity; "re-seal" = re-address after a patch. Distinct from the unrelated "seal" (close) sense.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **content address (CBOR / FNV-1a)** | A definition's identity: the **FNV-1a** hash of its **canonical CBOR** bytes — a deterministic binary encoding (RFC 8949 §4.2.1; ADR-0003 for the why). Identical definitions produce an identical address, so a projection computed once is provably correct on every surface. CBOR is the byte format; FNV-1a is the hash; together they are the address.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **host-wired**                      | A package's runtime status: it is instantiated and driven by a host (Astro, a worker, the edge), not imported and run by app code on its own. Contrast `standalone` (runs anywhere) and `standalone subsystem`. The status tag in PACKAGE-SURFACES.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **ECS**                             | Entity-Component-System — the composition model `@liteship/scene` uses: a scene is a world of entities, each carrying components, advanced by systems once per tick. Chosen so video / audio / transition / effect tracks compose without per-track glue (ADR-0009).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **DirtyFlags**                      | A bit-packed change-tracker on the hot path: up to 31 keys ride in one integer for allocation-free dirty checks; past 31 it falls back to a map. An instance of the "honest fallback past its regime" discipline (ADR-0002).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| **Effective candidate set**         | The responsive-media candidates safe to advertise under a request's Save-Data / DPR capabilities — the return of `selectCandidates(intent, caps)` in `@liteship/core` (`ResponsiveMediaCandidateSet`). THE one law every responsive-media output derives from: `src`, `srcset`, each `<source>`, the preload `imagesrcset`, the CSS `image-set()`, and the cache-key digest all enumerate it. Under Save-Data it is capped to the light/floor variant, so no artifact can advertise a heavier candidate (#140). "Effective" = after the capability cap, not the full authored set.                                                                                                                                                                                                                                                                                                   |

## Banned in marketing-style prose

_next-generation, leverage, robust, powerful, seamless, blazingly fast, cutting-edge, world-class, enterprise-grade, paradigm-shifting, game-changing, revolutionary, unleash, supercharge, harness the power of._ Replace with concrete behavior, or cut.

## Translator notes

A few terms in this corpus are polysemous; future i18n / machine-translation work should treat them as terms-of-art and pin the meaning rather than translate by surface form:

- **cast** — verb only, "project a definition into a target output surface" (CSS, GLSL, ARIA, etc.). Not the noun (theatrical cast) and not type-coercion (`as` casting). Always carries a target.
- **surface** — noun, "a runtime target the compiler emits to" (CSS surface, ARIA surface). Not the verb sense (something coming to attention).

## Verb grammar (authoring register)

One enforced verb per operation class (ADR-0046 / ADR-0051). The authoring surface reads as a grammar: **define** for immutable authored intent, **create** for runtime allocation, **computed** for a derived reactive value. Reach for the exact verb of the operation class rather than a synonym.

| Verb         | Meaning                                                            | Examples                                                                                                                                          |
| ------------ | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| **define**   | Create immutable authored intent — a declaration, evaluated later. | `defineBoundary`, `defineToken`, `defineTheme`, `defineStyle`, `defineConfig`, `defineAdaptive`, `defineQuantizer`                                |
| **create**   | Allocate stateful runtime behavior or a resource.                  | `createSignal`, `createCell`, `createLiveCell`, `createStore`, `createTimeline`, `createQuantizer`, `createLifetime`, `createGraphMutationClient` |
| **computed** | Derive a reactive value from other reactive sources.               | `computed(() => …)` — the derived cell                                                                                                            |
| **compile**  | Turn intent into a target artifact.                                | `CSSCompiler.compile`, `ARIACompiler.compile`, `compileReveal`                                                                                    |
| **decode**   | Validate/convert unknown data into typed data.                     | `decode`, `decodeLenient`                                                                                                                         |
| **resolve**  | Choose one concrete result from alternatives.                      | `resolveInitialState`, `chooseTier`                                                                                                               |
| **sample**   | Evaluate a time/index-dependent object.                            | `sampleProgram`                                                                                                                                   |
| **apply**    | Mutate a target using a validated operation.                       | `applyValidatedPatch`                                                                                                                             |
| **dispose**  | End owned runtime activity.                                        | direct `dispose()` / `Symbol.dispose` on a `Lifetime`-owning value                                                                                |

`define` returns a content-addressed declaration; nothing runs until a host evaluates it. `create` hands back a live object you `.read()`, `.subscribe()` to, and eventually `dispose()`. `computed` is the derived-reactive middle: a `create`-family value whose content is a pure function of its reactive inputs.

## Time semantics (two clocks, one rule)

A `timestamp` field names _which_ clock by its TYPE, never by the bare word. The two must never be confused — one bears identity, one does not.

| Term                                  | What it is                                                                                            | Contract                                                                                                                                                                                          |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **HLC** _(causal clock)_              | A hybrid logical clock `{wall_ms, counter, node_id}` (`@liteship/core` `HLC`).                        | Ordered + monotonic. **Included** in the receipt hash (`hashEnvelope`) and validated by the chain (`hlc_not_increasing`). Identity- and ordering-bearing. The capsule's `generated_at` is an HLC. |
| **WallClockTimestamp** _(wall clock)_ | A volatile ISO-8601 string stamped when a result is produced (`@liteship/core` `WallClockTimestamp`). | **Excluded** from `resultId`; never used for causal ordering. Provenance/display only. The `timestamp` on CLI / MCP / command receipts AND the `generatedAt` on report/artifact shapes are this.  |
| **media / performance time**          | Frame presentation time (µs/ms) or `performance.now()` relative ms.                                   | A different axis entirely — not a clock for identity or causality. Out of the HLC↔wall-clock split.                                                                                               |

Rule: an identity-adjacent command/result type must type a volatile stamp as `WallClockTimestamp` (or `HLC` for a causal one) — never a bare `timestamp: string`. Wiring a wall clock where an HLC belongs corrupts identity; the type names the contract so it can't happen silently.

The two generated-time names follow the same rule by their TYPE, not by being renamed:

- **`generated_at`** (snake_case) — the ship-capsule's HLC: causal, public, **identity-bearing** (hashed into the content address). Preserve it.
- **`generatedAt`** (camelCase) — report/artifact provenance: a volatile `WallClockTimestamp`. Stable field key, typed as the alias.

For _same-run coherence_ of gauntlet artifacts, the authoritative signal is **`gauntletRunId`** (a per-run UUID stamped into every artifact), NOT a `generatedAt` comparison. Any `Date.parse(generatedAt)` vs file-mtime check is a secondary wall-clock heuristic, not causal proof.

## Gauntlet & rigor engine (prose register)

The vocabulary of `@liteship/gauntlet` ([AUDIT.md](./AUDIT.md), [ADR-0023](./docs/adr/0023-gauntlet-rigor-engine.md)).

- **gauntlet** — the self-proving rigor engine; the gate set + the runner that qualifies them.
- **gate** — a `(context) → Finding[]` fitness function; earns blocking authority only by self-proof.
- **FactGate** — a gate whose decision is DATA over a declared FactPack (vs a closure), so it cannot read undeclared evidence ([ADR-0019](./docs/adr/0019-factgate-evidence-bound-gates.md)). Built by `defineFactGate`; the discriminant is an unforgeable `WeakSet`, not a string.
- **producer / kernel** — a FactGate's two halves: the producer does acquisition + normalization (host-side); the kernel is the bounded, data-only decision.
- **assurance level** — L0–L4; the hazard model that aims a gate's rigor. Authority decides the level, not folder location.
- **finding** — a gate's emitted result (`ruleId`, severity, level, location, remediation).
- **authority ratchet** — a gate is `advisory` until it self-proves red/green/mutation, then `blocking`. Authority is earned, not granted.
- **evidence digest** — the out-of-IR bytes a gate's verdict depends on, folded into the verdict-cache key. A FactGate derives it from its declared channels; a closure gate must author it.
- **green is not clean** — the standing discipline: a passing gauntlet means only that the gates that ran, on the surfaces they scanned, found nothing — never that the repo is correct.

## Drift check

After editing docs, run the sweep: mixed boundary verbs (_wire_ vs _rig_), banned words, accidental rename of `@liteship/*` or public APIs. The glossary holds; the prose comes back to it.
